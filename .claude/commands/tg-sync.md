---
description: Выгрузить дельту сообщений Telegram и обновить delta-log + context.md по каждому чату
---

Ты выполняешь sync отслеживаемых чатов Telegram. Делай строго по шагам.

## 1. Выгрузка дельты

Запусти скрипт из корня проекта **питоном из venv** (там стоит telethon;
системный python3 даст `ModuleNotFoundError: telethon`):

```bash
.venv/bin/python scripts/tg_sync.py
```

(Если просили только один чат — добавь `--only <name>`. Если venv нет —
сначала `python3 -m venv .venv && .venv/bin/pip install -r requirements.txt`.)

Скрипт разложит новые сообщения по недельным шардам `data/<name>/<YYYY-Www>.jsonl`
(ISO-неделя по дате сообщения, UTC), обновит `data/state.json` и запишет манифест
последней дельты в `data/<name>/.last_delta.json`.

Если скрипт упал на отсутствии `TG_API_ID/HASH` — сообщи пользователю, что нужно
заполнить `.env` (см. `.env.example`) и один раз залогиниться, и остановись.

## 2. Что именно новое

Прочитай `data/*/.last_delta.json`. Для каждой цели с `count > 0` новые сообщения
лежат в недельных шардах, перечисленных в поле `weeks` манифеста
(`data/<name>/<week>.jsonl`). Возьми из этих шардов строки с
`first_id <= id <= last_id`. Не перечитывай старые недели — только файлы из `weeks`.

Если по всем целям `count == 0` — скажи «новых сообщений нет» и заверши.

## 3. Обновить delta-log.md (слой 1 — хронология)

В `data/<name>/delta-log.md` допиши блок в начало секции записей:

```
## <дата ISO> — сообщения <first_id>..<last_id> (<count> шт.)
- <2–6 буллетов: что обсуждалось, кто что предложил/решил, упоминания пользователя>
- Решения: <если были — что и кто, со ссылкой #id>
- Вопросы/action items: <если были, с #id>
```

Опирайся ТОЛЬКО на текст дельты, не выдумывай. Указывай #id сообщений для ссылок.

## 4. Обновить context.md (слой 2 — живой общий контекст)

`data/<name>/context.md` — это backbone для будущего Q&A. Структура (создай файл
по этому шаблону, если его ещё нет; иначе аккуратно **смерджи** новую дельту,
не переписывая историю без причины):

```
# Контекст: <chat> / топик <topic>
_Обновлено: <дата>. Покрыто сообщений: id <min>..<max>._

## Участники
<имя (@username, from_id) — роль/зона ответственности, если выводится из переписки>

## Активные темы
<тема — текущий статус (1–2 строки), последнее движение #id>

## Принятые решения
<дата — решение — кто — #id>

## Открытые вопросы / ожидания
<вопрос — от кого ждут — #id>

## Договорённости и action items
<кто — что — срок (если есть) — #id>

## Глоссарий / сокращения
<термин — расшифровка>
```

Правила мерджа:
- новые решения/договорённости — ДОБАВЛЯЙ, старые не удаляй;
- если новая дельта закрывает открытый вопрос или меняет статус темы — обнови
  соответствующую строку, пометив, что изменилось;
- обнови строку `_Обновлено / Покрыто сообщений_`.

## 5. Вести журнал решений (decisions.jsonl)

`data/<name>/decisions.jsonl` — структурный append-only журнал решений и задач по
темам (его читают `/tg-topics` и `/tg-why`). Это тот же материал, что ты выделяешь
для `context.md`, но в запрашиваемом виде: тема + причина + следствие + статус.

Из новой дельты выдели **решения** и **задачи/action items** и дозапиши события.
Для каждого — стабильный `item_id` (kebab-slug сущности; для обновления уже
существующей сущности используй ТОТ ЖЕ `item_id`, чтобы события свернулись):

```bash
python3 - <<'PY'
import sys; sys.path.insert(0, "scripts")
import decisions
path = "data/<name>/decisions.jsonl"
decisions.append_event(path, {
    "ts": 0,                              # epoch сообщения-источника
    "item_id": "<kebab-slug>",            # та же сущность = тот же id
    "topic": "<тема>",
    "kind": "decision",                   # decision | task
    "statement": "<что решили / что за задача>",
    "why": "<причина: что в обсуждении к этому привело>",
    "effect": "<следствие/итог; опусти, пока неизвестно>",
    "owner": "<ответственный: имя/@handle; опусти, если не назначен>",
    "status": "decided",                  # open | decided | done | reversed
    "refs": [105],                        # id сообщений-источников
    "by": "tg-sync",
})
PY
```

Правила (как и для context.md):
- опирайся ТОЛЬКО на текст дельты, не выдумывай; всегда указывай `refs` (#id);
- НЕ переписывай прошлые события — фиксируй изменение НОВЫМ событием с тем же
  `item_id` (напр. статус `open` → `done`, добавление `effect`);
- если дельта ничего не решает и не порождает задач — журнал не трогай.

## 6. Отчёт пользователю

Кратко: по каждой цели — сколько новых, ключевые решения/упоминания, что
поменялось в context.md и какие записи добавлены в журнал решений. Без воды.

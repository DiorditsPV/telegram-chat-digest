## Проблема / цель
Имея журнал решений (#11), нужно быстро ответить: **почему** приняли решение по теме X и **что из
этого вышло** — связной цепочкой причина → решение → следствие, со ссылками `#id`. Это прямой ответ
на ключевую цель проекта.

## Поведение / UX
- `/tg-why <тема|вопрос>` находит релевантную сущность(и) в `data/<name>/decisions.jsonl`, печатает
  заголовок (тема/статус), причину (`why`), следствие (`effect`) и **хронологию событий** (как
  менялся статус во времени = причинно-следственная нить), затем при необходимости грепает шарды по
  `refs` для дословных цитат и отвечает пользователю.
- Если тема не найдена в журнале — деградирует к `/tg-ask`-подобному поиску по шардам.

## Затрагиваемые слои и файлы
- Логика (`scripts/decisions.py`): добавить `history(events, item_id)`, `search(items, query)`,
  `render_trace(events, item_id)`.
- Агентский слой: новая команда `.claude/commands/tg-why.md`.
- tests: `tests/test_tg_why.py`.

## Модель данных
Без новых артефактов — читаем `decisions.jsonl` (#11). Цепочка причина→следствие = хронологическая
последовательность событий одной сущности (`history`).

## Решения (с обоснованием)
- **Расширяем `decisions.py`, не плодим модуль** — `tg-why` это представление поверх того же журнала.
- **`render_trace` отдельно от `render_board`** — доска отвечает «что есть», trace — «почему и что вышло».
- **`search` по подстроке (casefold)** — просто и без зависимостей; ранжирование не усложняем (цель —
  лёгкость). Альтернатива (нечёткий матч/индекс) избыточна для масштаба.

# Скилл: проектирование фичи (design → спека, без кода)

Результат — **спека** в `.claude/features/<slug>.md`, которую затем реализует **feature-build**.
Код здесь НЕ пишем. Цель — продумать фичу так, чтобы реализация была механической.

Этот скилл **проектно-независимый**: всё, что специфично для репозитория (карта файлов,
конвенции, как проверять), берётся из конфига `.claude/feature-loop.md`.

## 0. Загрузить контекст проекта (ground truth, через `cat`, НЕ Read — кэш устаревает)
- `cat .claude/feature-loop.md` — карта проекта, конвенции, секция Verify, настройки
  (`branch_prefix`, `default_branch`, `ideas_file`, `specs_dir`, …). **Без этого файла не
  проектируй**: предложи установить фреймворк (`install.sh`) и заполнить конфиг.
- `cat <ideas_file>` (по умолчанию `FEATURE_IDEAS.md`) — что хотим.
- `cat <features_catalog>` (`FEATURES.md`) и `ls <specs_dir>` — что **уже** сделано/спроектировано,
  чтобы НЕ дублировать.
- Реальный код смотри через `cat`/`grep` по файлам из «Карты проекта». Не выдумывай API — сверяйся с кодом.

## 1. Взять идею + режим
- **Интерактивно** (просьба пользователя): прочти запрос; при неоднозначности, меняющей реализацию
  (формат / место / поведение / объём) — задай 2–4 уточняющих вопроса (AskUserQuestion, по возможности
  с превью). Не уточняй то, что выбирается разумным дефолтом — выбери и зафиксируй в спеке.
- **Автономно** (из лупа, см. `.claude/AUTODEV.md`): возьми следующую `[ ]`-идею из `ideas_file`;
  если бэклог пуст — придумай 1 новую (из пробелов/UX) и допиши её в `ideas_file`. Вопросы не задавай.

## 2. Решения по дизайну
- **Слои/файлы**: какие модули из «Карты проекта» затронуты и какие конкретно файлы.
- **Модель данных**: новые поля/сущности? где живут, нужна ли миграция, какие инварианты схемы
  затрагиваются (см. «Конвенции и грабли» в конфиге).
- **UX/поведение**: как выглядит и управляется; вписать в существующие паттерны проекта; крайние случаи.
- **Совместимость**: что не сломать; что добавить в тесты/проверки (опираясь на секцию Verify конфига).
- Спорные решения — фиксируй с обоснованием (выбранный вариант + почему + отвергнутые альтернативы).

## 3. Записать спеку
Создай `.claude/features/<slug>.md` (slug — kebab-case фичи). Скелет — в `.claude/features/_TEMPLATE.md`;
скопируй и заполни. Обязательный frontmatter:

```yaml
---
slug: <kebab-slug>
title: <короткое название фичи>
status: designed        # designed -> building -> done
created: <YYYY-MM-DD>
branch: feature/<kebab-slug>   # branch_prefix из конфига
verify: —               # заполняет feature-build: pass | fail
review: —               # заполняет feature-build: ok | <кратко замечания>
---
```

Разделы тела: **Проблема/цель**, **Поведение/UX**, **Затрагиваемые слои и файлы**, **Модель данных**,
**Решения (с обоснованием)**, **План реализации (чеклист для feature-build)**, **Тесты/приёмка**,
**Риски/открытые вопросы**. План — нумерованный чеклист `[ ]` атомарных шагов (по возможности — с
сообщением коммита на шаг), чтобы build шёл по нему механически.

## 4. Отдать
Кратко перескажи спеку пользователю (1–2 абзаца) и укажи путь файла. Предложи запустить **feature-build**
для реализации. Код не пиши.

# Скилл: реализация фичи (build по спеке, в ветке фичи)

Реализует то, что спроектировал **feature-design**. Код пишем, но строго по спеке и в ветке фичи.
Скилл **проектно-независимый**: настройки, конвенции и команды проверки берёт из `.claude/feature-loop.md`.

## 0. Загрузить контекст и спеку (через `cat`, НЕ Read)
- `cat .claude/feature-loop.md` — настройки (`branch_prefix`, `default_branch`, `specs_dir`,
  `commit_trailer`, `reused_skills`), карта проекта, конвенции, секция **Verify**.
- Найди спеку в `specs_dir`: `ls .claude/features/` → возьми названную пользователем, иначе последнюю
  со `status: designed|building`. Прочти через `cat`. Спеки нет — предложи сначала **feature-design**,
  не выдумывай фичу сам.

## 1. Ветка фичи (ОБЯЗАТЕЛЬНО — не работать в default-ветке)
- `git rev-parse --is-inside-work-tree`. Если не git — предупреди и спроси: `git init` или продолжать
  без ветки (по умолчанию НЕ продолжать молча).
- `git status --porcelain`. Есть чужие незакоммиченные правки — спроси, как быть (не затирай).
- Создай/переключись на ветку от актуальной `default_branch`:
  ```bash
  git switch -c <branch_prefix><slug> 2>/dev/null || git switch <branch_prefix><slug>
  ```
  Никогда не коммить код фичи прямо в `default_branch`.
- В спеке выставь `status: building`.

## 2. Реализовать по плану — ПЕРЕИСПОЛЬЗУЯ скиллы проекта
Иди по чеклисту «План реализации» из спеки, отмечая шаги `[x]`. Не дублируй логику доменных скиллов —
**вызывай их через Skill-инструмент** (список и «когда» — в `reused_skills` конфига). Соблюдай раздел
«Конвенции и грабли» конфига (формат данных, запреты, паттерны, ground-truth-источник).

## 3. Гейт качества (Verify + code-review) — ОБЯЗАТЕЛЕН
1. Прогони команды из секции **Verify** конфига (+ проверки из «Тесты/приёмка» спеки), выбирая нужные по
   правилу «когда что нужно». Если правка переименовала сущность, на которую опирается тест/smoke — обнови тест.
2. Прогони **code-review** по диффу ветки; краткое резюме замечаний запиши в спеку (раздел в конце + поле
   frontmatter `review:`). Поле `verify:` → `pass`/`fail`.
3. **Красный verify = не оставлять полуфабрикат:** либо доведи до зелёного, либо откати ветку
   (`git switch <default_branch>` + удали ветку), пометь идею в `ideas_file` проблемной и не пиши в каталог.

## 4. Коммит на ветке (без пуша)
```bash
git add -A && git commit -m "feat(<slug>): <title из спеки>"
```
Если `commit_trailer` в конфиге задан — добавь его строкой в конец сообщения коммита.
**НЕ пушить и НЕ открывать PR без явной просьбы пользователя.**

## 5. Обновить ledger — НА `default_branch` (критично для идемпотентности!)
Код фичи остаётся на ветке. Но статус/каталог надо писать **на `default_branch`**, иначе следующий цикл,
читая её, возьмёт ту же идею снова (дубль).
1. `git switch <default_branch>` (ветку фичи не мёржим, не удаляем).
2. На `default_branch`: в спеке `status: done` + `verify`/`review`/`branch`; пометь идею в `ideas_file`
   (`- [ ] <slug> …` → `- [x] <slug> (<branch_prefix><slug>)`); пересобери каталог
   `python3 .claude/skills/feature-build/catalog.py` (обновит `features_catalog`).
3. Коммит ledger-файлов **на default_branch** (docs-only, БЕЗ push):
   ```bash
   git add <ideas_file> .claude/features/<slug>.md <features_catalog> && git commit -m "chore(<slug>): mark done + catalog"
   ```
   (+ `commit_trailer`, если задан).

## 6. Отчитаться
Кратко: что реализовано по слоям, результат Verify + code-review, имя ветки, переиспользованные скиллы,
что осталось (push/PR/merge — за пользователем). В автономном режиме — следуй СТОП-условиям `.claude/AUTODEV.md`.

# Скилл: аудит портфеля фич и архитектуры

Взгляд сверху на всё сразу — в отличие от `feature-design` (одна фича). Код не пишет: только
анализирует и предлагает. Результат — отчёт в чат и (по желанию) `FEATURE_REVIEW.md`.

## 0. Собрать картину (через `cat`/`ls`, НЕ Read)
- `python3 .claude/skills/feature-build/catalog.py` — освежить `FEATURES.md` (источник истины — спеки).
- `cat .claude/feature-loop.md` (карта проекта, конвенции, Verify, Deploy), `cat FEATURE_IDEAS.md`,
  `cat FEATURES.md`, `ls .claude/features/`.
- Реальную структуру кода смотри через `git ls-files`/`ls`/`grep` по слоям из «Карты проекта».

## 1. Портфельный анализ (реализованные + в работе + бэклог)
- **Покрытие по слоям:** разнеси фичи (done/shipped + `[ ]`-идеи) по слоям из карты проекта. Где густо,
  где пусто (слой без единой фичи/идеи = потенциальный пробел или зона риска).
- **Дубли/перекрытия:** идеи/спеки, делающие близкое; предложи объединить или развести.
- **Зрелость:** сколько `shipped` vs `done`-кандидатов висит на ветках без выкатки (накопление долга ревью).

## 2. Сверка с архитектурой
- Какие фичи тянут за собой **крупный рефактор / смену модели данных** (по «Конвенциям и граблям») —
  пометь как архитектурно-значимые, предложи порядок (фундамент раньше надстроек).
- Противоречия между фичами (одна ломает инвариант, нужный другой) — выяви заранее.
- Технический долг, который блокирует бэклог — назови явно.

## 3. Дрейф (консистентность ledger)
- спеки `done`/`shipped`, у которых **нет ветки** `<branch>` (или наоборот — ветка есть, спека `designed`);
- идеи `[x]` в `FEATURE_IDEAS.md` **без** соответствующей спеки; спеки без записи-идеи;
- `FEATURES.md` разошёлся с frontmatter спек (после `catalog.py` не должен — если разошёлся, отметь).

## 4. Актуальность скиллов и доков (поддержка фреймворка)
- **`feature-loop.md` ↔ реальность:** «Карта проекта» ещё описывает текущую структуру? (сверь с `git ls-files`).
  Пути/слои не переехали?
- **Verify-команды ещё валидны?** Проверь, что инструменты/скрипты из секции Verify существуют
  (`command -v`, наличие файлов-скриптов) — мёртвая команда в гейте опаснее всего.
- **Deploy-секция** соответствует тому, как реально катится проект?
- **AUTODEV.md / README.md** не противоречат текущим настройкам (`branch_prefix`, `default_branch`).
- Доменные `reused_skills` ещё существуют и описаны верно?
Найденные расхождения — конкретным списком «файл → что поправить» (правки доков можешь применить сразу,
если они очевидны и не меняют поведение; иначе предложи).

## 5. Вывод
- **Отчёт** (в чат; по желанию — `FEATURE_REVIEW.md`): покрытие, пробелы, дубли, дрейф, риски, состояние доков.
- **Приоритезированный список «что дальше»** (3–7 пунктов): какие идеи проектировать первыми и почему;
  что отрефакторить; что выкатить; что починить в доках/Verify. Новые идеи допиши в `FEATURE_IDEAS.md` (`[ ]`).
- Ничего не выдумывай про код — все выводы со ссылкой на файлы/спеки/команды.
